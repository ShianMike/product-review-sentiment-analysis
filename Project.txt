I. INTRODUCTION

1.1. Background of the Study
Customer product reviews are a major source of information that influences purchase decisions and reflects customer satisfaction. Modern e-commerce and retail environments generate a large volume of reviews, making it difficult for businesses to manually read, summarize, and identify recurring concerns quickly. Consumer research has shown that reviews are widely used and considered essential in purchase decisions, highlighting the importance of extracting insights from review content efficiently (PowerReviews, 2023).

Data analytics is suitable for this problem because product reviews are unstructured text that can be transformed into structured insights through natural language processing (NLP) and machine learning. Sentiment analysis helps classify reviews into sentiment categories (positive, neutral, negative), while text mining can extract themes such as common complaints and praises. This enables businesses to track customer satisfaction trends over time and prioritize improvement areas based on evidence rather than manual guessing (Tan et al., 2023; Mao et al., 2024).

Despite the availability of reviews, current challenges include the time required for manual analysis, inconsistent interpretation across people, and delayed detection of recurring product issues. These inefficiencies can negatively affect decision-making related to product improvement, customer support response, and quality assurance.
1.2. Problem Statement
Businesses and sellers face difficulty in efficiently interpreting large volumes of product review text to identify overall customer sentiment and recurring product issues. Reviews often contain valuable feedback, but manual analysis is time-consuming and may cause delays in detecting common complaints that require immediate action. Since reviews strongly influence consumer decisions, the inability to quickly extract actionable insights can reduce customer satisfaction and competitiveness (PowerReviews, 2023).
This project addresses the need for a web-based system that can automatically analyze product review datasets, classify sentiment, summarize key themes, and present insights through an interactive dashboard for decision support.
 
II. OBJECTIVES OF THE STUDY

2.1. General Objective

To develop a web-based product review analytics dashboard that transforms customer review text into overall sentiment insights, aspect-based sentiment insights, and theme summaries to support data-driven decision-making.

2.2. Specific Objectives

1.	To collect and preprocess product review datasets by validating required fields and performing text cleaning, tokenization, lemmatization, and normalization, while preserving negation words that carry sentiment meaning.
2.	To apply appropriate data mining techniques, including TF-IDF based sentiment classification (Logistic Regression baseline) and rule-based aspect-based sentiment analysis (ABSA), to label reviews as positive, neutral, or negative and compute sentiment polarity for key product aspects.
3.	To extract and summarize key product aspects and themes from reviews (e.g., quality, price, delivery, taste, service, appearance, usability) and identify frequent complaints and praises per aspect through keyword and phrase extraction.
4.	To develop a functional web-based analytics dashboard that presents sentiment distribution, aspect-level breakdowns, theme summaries, trend charts (when timestamps are available), and product-level comparisons with filtering, searching, and export features.
5.	To evaluate the performance of the sentiment classification model using accuracy, precision, recall, and F1-score, and to compare alternative classifier candidates (e.g., Linear SVM, Naive Bayes, SGD) on the same train/test split to justify the production model choice.
6.	To support a single-review prediction interface and a model information page so that the trained classifier can be tested and inspected independently of a full dataset upload.

III. SCOPE AND DELIMITATIONS

3.1.  Scope

•	The system will accept product review datasets via CSV/Excel upload (.csv, .xlsx, .xls) and will process review text with optional fields such as rating, date, product identifier, and summary, using auto-detection so different upload schemas work without manual column mapping.
•	The system will apply text preprocessing, sentiment classification, and aspect-based sentiment analysis to produce overall sentiment labels with confidence scores and aspect-level sentiment results across seven aspect categories: quality, price, delivery, taste, service, appearance, and usability.
•	The system will extract keywords (TF-IDF) and recurring phrases (bigrams/trigrams) to summarize common complaints and praise themes, including sentiment-bucketed word clouds.
•	The system will aggregate sentiment, themes, and trends at the product level when product identifiers are present, allowing aspect, theme, trend, and review views to be scoped to a single product.
•	The system will display results through a web dashboard with visualizations, filtering (product, sentiment, date range, keyword/theme), searching, sorting, pagination, and export features (CSV processed reviews, JSON dashboard summary, dedicated aspect-analytics CSV/JSON, and filtered-review CSV).
•	The system will additionally provide a single-review prediction interface for testing the trained classifier and a model information page for inspecting evaluation metrics and candidate model comparisons.
•	The system will persist completed analyses as backend saved projects so users can reload previous dashboard results or delete stored analyses from the upload page.
•	The intended users include sellers, product managers, and customer support teams who need faster review insights for decision-making.

3.2.  Delimitations

•	The project will not implement real-time review scraping or direct integration with e-commerce platforms.
•	The system will be designed for batch analysis of uploaded datasets rather than continuous streaming data.
•	Advanced NLP tasks such as sarcasm detection and deep contextual understanding will not be prioritized in the prototype.
•	Multilingual sentiment analysis will not be a main focus unless included as an extension.
•	No database will be used in the prototype; outputs will be file-based through exported reports/results and backend saved-project JSON files.


IV. REVIEW OF RELATED LITERATURE AND SYSTEMS

4.1. Related Studies

Sentiment analysis is a well-established area in NLP that focuses on extracting and classifying opinions expressed in text into polarity categories such as positive and negative. Early work showed that review polarity could be inferred from textual orientation cues (Turney, 2002). Liu (2012) explains that sentiment analysis (opinion mining) aims to identify and analyze opinions, sentiments, and evaluations from textual data, while Pang and Lee (2008) describe sentiment analysis as a major NLP task for organizing subjective text. This makes the technique suitable for product reviews, where customer opinions are the main source of insight.

Recent literature shows that sentiment analysis pipelines typically include preprocessing, feature representation (e.g., TF-IDF or embeddings), and classification using machine learning, deep learning, or ensemble learning approaches (Medhat et al., 2014; Tan et al., 2023; Mao et al., 2024). TF-IDF remains a common and interpretable baseline feature weighting approach for text mining and information retrieval (Salton & Buckley, 1988; Manning et al., 2008). For product reviews, transformer-based, deep learning, and hybrid approaches are widely studied due to their ability to capture context and improve classification performance compared with simpler baselines (Zhang et al., 2018; Chen, 2024; Ghatora et al., 2024; Daza et al., 2024; Li et al., 2024).

Recent work in e-commerce review analytics highlights the value of extracting not only overall sentiment but also fine-grained, aspect-based sentiment to better understand drivers of customer satisfaction (Davoodi et al., 2025; Wankhade et al., 2024). Earlier product-review summarization work also showed that identifying product features and their associated opinions helps users understand strengths and weaknesses without reading every review manually (Hu & Liu, 2004). Aspect-based sentiment analysis benchmarks such as SemEval further formalized aspect category detection and polarity classification as important subtasks (Pontiki et al., 2014).

4.2. Related Systems

Existing tools provide review analytics, but many require paid subscriptions or platform-specific integration. Large e-commerce platforms are also beginning to automate review summarization, showing the practical value of reducing long review lists into concise insights (Hadero, 2023). General analytics dashboards often focus on numerical KPIs and may not directly provide sentiment insights from unstructured text. This proposed system focuses on a lightweight, upload-based workflow for sentiment analysis and thematic summarization, suitable for small businesses or academic prototyping, while emphasizing interpretability and evaluability.

V. CONCEPTUAL FRAMEWORK
Input
The system accepts a customer review dataset (CSV/Excel) containing review text and optional fields such as rating, date, product identifier, and summary. Column names are auto-detected from common naming conventions.
Process
The system validates the dataset, preprocesses the text (cleaning, tokenization, lemmatization with negation preservation), performs overall sentiment classification using a TF-IDF + Logistic Regression model with rule-based confidence calibration, and runs rule-based aspect-based sentiment analysis (quality, price, delivery, taste, service, appearance, usability) with TextBlob polarity scoring per aspect. Themes and recurring phrases are extracted using TF-IDF keyword ranking and n-gram frequency counts. When the dataset includes dates, monthly sentiment trends and per-aspect monthly trends are computed. When product identifiers exist, results are also aggregated per product and made available for on-demand product-scoped drill-down through a backend re-aggregation endpoint.
Output
The system generates a web dashboard with five analysis views — Overview, Aspects, Themes, Trends, and Reviews — that display overall sentiment distribution, rating distribution, product-level sentiment comparison, aspect-level sentiment breakdowns and drill-downs, sentiment trends (overall, per-product, and per-aspect), keyword and phrase summaries, praise/complaint word clouds, and a searchable/filterable review table with a review detail modal. The dashboard also exposes a single-review Test Prediction page and a Model Info page (live evaluation metrics, confusion matrix, classification report, and multi-model comparison results), plus filtering and multi-format export features.

VI. METHODOLOGY

6.1. Data Source and Description
•	Dataset Source: Public datasets from repositories such as Kaggle.
•	Type of Data: Primarily unstructured text (reviews) with optional structured columns (rating, date, product/category).
•	Attributes/Features: Review text, predicted sentiment label, sentiment score, extracted keywords/themes, and optional metadata.

6.2. Data Preprocessing

•	Text cleaning (HTML/URL/email stripping, lowercasing, punctuation removal, whitespace normalization, and zero-width character removal).
•	Tokenization, lemmatization, and stopword removal using standard NLP preprocessing practices, with negation words intentionally preserved so that polarity-flipping context (e.g., "not good") is not discarded (Bird et al., 2009).
•	Handling missing values by removing empty reviews and marking missing optional metadata fields so downstream steps degrade gracefully.
•	Auto-detection of common column names (review text, rating, date, product identifier, summary) so different upload schemas work without manual mapping.
•	Rating-to-sentiment label derivation during training data preparation (1–2 = negative, 3 = neutral, 4–5 = positive), with optional hybrid TextBlob re-labeling on ambiguous mid-rating rows to improve neutral-class quality.
•	Transforming text into numerical features using TF-IDF vectors (unigrams + bigrams) for classification and theme extraction, following established term-weighting practices in text retrieval and classification (Salton & Buckley, 1988; Manning et al., 2008).

6.3. Data Analytics Techniques

•	Chosen Technique: Sentiment Classification and Aspect-Based Sentiment Analysis (ABSA).
•	Justification: Overall sentiment supports quick satisfaction monitoring, while ABSA identifies which product aspects drive negative/positive sentiment for better prioritization (Tan et al., 2023; Mao et al., 2024).
•	Sentiment Model: Logistic Regression trained on TF-IDF unigrams + bigrams (up to 50,000 features) with class-weight rebalancing for the neutral class. A rule-based calibration layer using positive, negative, and neutral cue lexicons adjusts low-confidence single-text predictions.
•	ABSA Approach: Rule-based aspect detection using boundary-aware compiled regex patterns over per-aspect keyword lexicons, combined with sentence-level TextBlob polarity scoring. The rule-based design is intentional to keep the prototype transparent and easy to explain during evaluation, while still following the core idea of linking product aspects/features to opinion polarity (Hu & Liu, 2004; Pontiki et al., 2014; Loria, 2018).
•	Supporting Technique: Text mining for aspect/theme extraction and keyword/phrase summarization, including TF-IDF keyword ranking, CountVectorizer phrase extraction (bigrams/trigrams), sentiment-bucketed praise/complaint keyword lists, and word-cloud frequency data per sentiment class.
•	Product-Level Aggregation: Per-product sentiment summaries with an attention-scoring heuristic that combines negative share, neutral share, and average rating to flag products that may need seller attention.
•	Model Comparison: An offline comparison utility evaluates Logistic Regression, Linear SVM, Multinomial Naive Bayes, Complement Naive Bayes, and SGD on the same train/test split to justify the production model and surface candidate alternatives in the Model Info page. These candidate models are implemented using scikit-learn, a standard machine-learning library for Python (Pedregosa et al., 2011).

6.3.1. Model Comparison Results and Final Model Choice

The project compares five classical machine-learning text classifiers using the same TF-IDF unigram + bigram feature representation and the same train/test split. Macro F1 is treated as the main ranking metric because review sentiment datasets may be class-imbalanced, especially around the neutral class.

Model Comparison Summary:
•	Rank 1 – Logistic Regression: Accuracy = 0.8936, Macro Precision = 0.7396, Macro Recall = 0.7097, Macro F1 = 0.7237, Fit/Predict Time = 10.406 seconds.
•	Rank 2 – Linear SVM: Accuracy = 0.8871, Macro Precision = 0.7141, Macro Recall = 0.7178, Macro F1 = 0.7159, Fit/Predict Time = 7.621 seconds.
•	Rank 3 – SGD Logistic Classifier: Accuracy = 0.8719, Macro Precision = 0.6992, Macro Recall = 0.5998, Macro F1 = 0.6348, Fit/Predict Time = 1.207 seconds.
•	Rank 4 – Complement Naive Bayes: Accuracy = 0.8267, Macro Precision = 0.5978, Macro Recall = 0.6753, Macro F1 = 0.6183, Fit/Predict Time = 0.300 seconds.
•	Rank 5 – Multinomial Naive Bayes: Accuracy = 0.8566, Macro Precision = 0.8072, Macro Recall = 0.4746, Macro F1 = 0.5103, Fit/Predict Time = 0.280 seconds.

The final model selected for the application is Logistic Regression with TF-IDF unigrams + bigrams. It was chosen because it achieved the best macro F1 and the highest overall accuracy while remaining fast enough for local prototype use, interpretable as a linear baseline, easy to persist with joblib, and compatible with probability outputs needed for confidence scores in the dashboard.

VII. SYSTEM ARCHITECTURE AND DESIGN

7.1. System Architecture

•	Frontend: React single-page application (Create React App) rendering the dashboard UI for upload, filters, visualizations (Recharts), and exports. Custom CSS is used for styling; light/dark themes are supported via a CSS variable system.
•	Backend API: Flask REST server (with Flask-CORS) that exposes endpoints for health check, upload/analyze, asynchronous job start and status polling, exports (CSV/JSON/aspect-specific), saved project list/load/delete actions, generated-file cleanup, single-review prediction, product-scoped analysis, full reviews-table retrieval, and model information.
•	Analytics Engine: Python pipeline split into numbered modules covering preprocessing, sentiment classification, ABSA, aspect themes, aspect trends, global themes, monthly trends, product summaries, reviews-table assembly, and model comparison.
•	Storage: No database. File-based inputs and outputs are stored locally in uploads/, exports/, and projects/ folders. The projects/ folder stores completed analysis payloads so users can reload or delete saved backend analyses from the upload page.

7.2. Functional Requirements

•	The system shall allow users to upload CSV/Excel review datasets (.csv, .xlsx, .xls) and shall reject other file types with a clear error message.
•	The system shall validate required columns (at minimum: review text) and shall auto-detect optional columns such as rating, date, product identifier, and summary.
•	The system shall preprocess text and perform sentiment classification (positive/neutral/negative with confidence) and aspect-based sentiment analysis across the defined aspect categories.
•	The system shall compute overall, per-aspect, and per-product sentiment summaries, extract key themes (keywords + recurring phrases), and build word-cloud frequency data per sentiment class.
•	The system shall provide an asynchronous upload-and-analyze workflow with a job ID, status polling, and progress messages so users see pipeline stage updates instead of a frozen UI.
•	The system shall present results in a dashboard with five sections — Overview, Aspects, Themes, Trends, Reviews — and shall support filtering by product, sentiment, date range, and keyword/theme, as well as searching, sorting, pagination, and exports (processed CSV, dashboard JSON, aspect CSV/JSON, and a filtered-review CSV from the Reviews tab).
•	The system shall provide a Review Details view (modal) showing the full review text, summary (if available), predicted sentiment, confidence, rating, date, product identifier, and detected aspects with per-aspect sentiment labels.
•	The system shall expose a single-review prediction endpoint and a Test Prediction page for testing the classifier outside of dataset uploads.
•	The system shall expose backend health status, evaluation metrics (accuracy, precision, recall, F1, confusion matrix, classification report), and a multi-model comparison summary through a Model Info page.
•	The system shall persist completed analyses as backend saved projects and shall allow users to load or delete those saved projects from the upload page.
•	The system shall enforce configurable upload size limits and shall provide generated-file cleanup controls for uploads, exports, and saved project files.

7.3. Non-Functional Requirements

•	Performance: The system should process a moderate dataset within reasonable time for classroom-scale use.
•	Usability: The interface should be simple with clear labels, charts, and filters for non-technical users.
•	Security (basic): Uploaded datasets should not be publicly exposed; the system should avoid storing sensitive identifiers and should support local/offline use if required.
•	Reliability and maintainability: The backend should return clear error messages, use logging for failures, isolate storage utilities, and include unit tests for key pipeline and storage behaviors.

VIII. TOOLS AND TECHNOLOGIES

•	Frontend: React (Create React App), Recharts (charts), lucide-react (icons), react-dropzone (file selection), axios (HTTP client). Styling is handled with hand-written CSS using CSS custom properties for theming (light/dark).
•	Backend / API: Python with Flask and Flask-CORS exposing REST endpoints for upload, asynchronous analysis, exports, saved projects, storage cleanup, single-review prediction, product-scoped drill-down, and model information.
•	Analytics / NLP / ML (Python): pandas, numpy, scikit-learn (TfidfVectorizer, CountVectorizer, LogisticRegression, LinearSVC, MultinomialNB, ComplementNB, SGDClassifier), NLTK (stopwords, WordNetLemmatizer), TextBlob (polarity scoring used in ABSA and hybrid labeling), and joblib (model persistence).
•	Storage: Local filesystem storage is used for temporary uploads, generated exports, and saved analysis project JSON files (uploads/, exports/, and projects/ folders). Saved Projects provide the reload and delete workflow for completed analyses without requiring a database.
•	Deployment Environment: Web-based UI served by the React development server, talking to the Flask development server for the prototype. Designed for local execution; CORS origins, upload size limits, generated-file count limits, and storage retention hours are configurable through environment variables.
